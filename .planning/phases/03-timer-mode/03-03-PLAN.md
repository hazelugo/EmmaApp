# Plan 3.3: App Orchestration & Wiring

## Objective
Wire the `useTimer` composable and related UI components into `App.vue` so the actual game logic can switch between standard mode and timer mode.

## Context
- .planning/phases/03-timer-mode/03-CONTEXT.md
- src/App.vue

## Tasks

<task type="auto">
  <name>Integrate useTimer and UI into App.vue</name>
  <files>
    src/App.vue
  </files>
  <action>
    - Import `useTimer` and `TimerResultsOverlay`.
    - Import `generateOperands` from `src/utils/mathUtils.js`.
    - Setup state: define `isTimerMode` ref. Manage a local `handleSprintStart()` which invokes `timer.startTimer(handleSprintEnd)` and sets `isTimerMode=true`, immediately generates a timer-specific problem (`+` or `-` only), resets `answer.value`.
    - Pass `isTimerMode` and `timer.timeLeft` to `<ScoreHeader>`.
    - Place `<TimerResultsOverlay>` component, wired to timer state.
  </action>
  <verify>npm run build</verify>
  <done>Dependencies and UI elements are bound to state.</done>
</task>

<task type="auto">
  <name>Handle Timer problem generation and check answer</name>
  <files>
    src/App.vue
  </files>
  <action>
    - In `onSubmit()`, fork logic: if NOT `isTimerMode`, use standard `checkAnswer()`.
    - If `isTimerMode`, implement timer-specific answer checking: check `Number(answer.value)` against `currentProblem.a +/- b`.
    - If correct: play beep, increment `timer.correctCount`, add 1 to `timer.coins`, and immediately generate next problem (no waiting!). Randomly select `+` or `-` constraint. Use `difficulty.maxOperandByOperator` for bounds.
    - If wrong: clear input `answer.value = ''`, play wrong sound, don't penalize.
    - When `handleSprintEnd` fires: close timer mode, add `timer.coins` to main game `stars.value` via `stars.value += timer.coins` (saving to localStorage), discard in-progress answer (force `answer.value=''` and maybe restore normal game problem).
  </action>
  <verify>npm run build</verify>
  <done>Timer Mode rules correctly implemented without corrupting standard mode.</done>
</task>

## Success Criteria
- [ ] User can click 'Sprint' to start a 60s timer session.
- [ ] Problems rapidly refresh upon getting them correct (skip the long delay).
- [ ] Only '+' and '-' problems appear during Sprint mode.
- [ ] When time expires, Sprint mode ends, UI overlay shows results, and coins are added to main pool.
